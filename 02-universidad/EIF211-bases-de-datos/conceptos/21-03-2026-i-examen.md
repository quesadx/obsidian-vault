## 1. SQL (DDL, DML, DCL)

El SQL es un lenguaje tanto de manipulación, control y definición de datos. Se subdivide en categorías:
- `DDL (Data Definition Language)`: Se encarga de definir las estructuras de datos a utilizar.
- `DML (Data Manipulation Language)`: Se encarga de la manipulación de datos.
- `DCL (Data Control Language)`: Se encarga de la capa de control (seguridad) sobre dichos datos.

## 2. Modelado de datos

El modelado de datos corresponde a la arquitectura asociada a la información. Proporciona seguridad, integridad y una visión clara del mapa antes del código.

### 2.1 Modelo entidad-relación (ER)

Corresponde a una estructuración lógica de la información antes de una implementación técnica

### 2.2 SQL (Structured Query Language)

Es el lenguaje estándar utilizado para interactuar con RBDMS's (Relational Database Management System). Son los que se fundamentan en tablas y columnas.

```SQL
BEGIN;

UPDATE Cuenta SET saldo = saldo - 100 WHERE id = 1;
UPDATE Cuenta SET saldo = saldo + 100 WHERE id = 2;

COMMIT;
```

`BEGIN` inicia la transacción, y `COMMIT` la confirma. Si algo falla, se puede usar `ROLLBACK`.

### 2.3 Bases de datos no-relacionales

Son sistemas de almacenamiento que renuncian a las tablas y esquemas rígidos en favor de la flexibilidad y el rendimiento masivo

Resaltan en:

1. **Big Data:** Redes sociales y analítica masiva.
2. **Datos dinámicos:** Información que cambian continuamente.
3. **Schema-less:** Sin estructura predefinida.
4. **Escalabilidad:** Distribución en múltiples servidores.

Los sistemas de bases de datos no-relacionales resaltan por su **escalabilidad horizontal**, **Sharding (división de cargas para cargas masivas)**, **Replicación automática** y **Flexibilidad**.

## MongoDB

Existen diferentes modelos de almacenamiento en bases de datos no relacionales:
1. **Key-Value Store**: clave -> valor. Acceso a BLOBs mediante clave única. Ideal para imágenes, audio, video. Resulta escalable y simple.
2. **Column-Family**: Almacena blobs versionados. Organiza datos por familias de columnas, no filas rígidas. Tiene columnas dinámicas (un usuario puede, en vez de edad, tener un atributo ciudad).
3. **Graph**: Datos almacenados en nodos, relaciones y propiedades. Uso ideal donde la relación es la clave (Redes Sociales).
4. **Document-Store**: Los datos se guardan como una unidad completa, cualquier item dentro de la unidad (documento). **Esta es la picante, la que se usa en este contexto.**

## Modelo Entidad-Relación (ER)

El modelo ER es intuitivo y gráfico. Sirve como esquema conceptual independiente de la tecnología.

### Participación y límites (min, max)

Corresponde a la pequeña anotación que va al lado de la entidad indicando la cantidad mínima de ocurrencias de la entidad (mínimas y máximas) con respecto a la relación.

- **Líneas dobles** **-> Participación total:** Significa que todas las instancias de esa entidad deben participar obligatoriamente en la relación.
- **Bordes dobles en entidad -> entidad débil (dependiente):** Su existencia depende totalmente de la entidad propietaria. Su clave primaria se conforma uniendo la clave primaria de la entidad fuerte + discriminador (algo que la hace única, e.g., un nro_item); resulta que esto es una clave primaria compuesta. Este discriminador se anota con **linea punteada - - - - -**.

### Modelo Enhanced Entity-Relationship (ERE)
supertipos
Es una extensión de ER que agrega, además de un par de cosas ya mencionadas en clase, asuntos como las categorías o supertipos, junto a especialización o generalización (herencia).
