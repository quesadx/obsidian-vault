# Resumen conceptos bases de datos

## 1. Arquitectura lógica

### 1.1 Schema
- Contenedor lógico que agrupa objetos de la base de datos (tablas, vistas, índices, etc.).
- Sirve para organización, seguridad y separación de nombres de objetos.

### 1.2 Tablespace
- Unidad lógica de almacenamiento dentro de un DBMS.
- Agrupa archivos físicos en disco (datafiles) para almacenar segmentos.
- Define en qué parte del disco viven los datos.

### 1.3 Índice
- Estructura de datos que acelera búsqueda de filas en una tabla.
- Permite acceso rápido mediante clave, evitando full table scans.
- Tipos comunes: B-tree, hash, índices compuestos, index-organized tables.

---

## 2. Arquitectura de memoria (Oracle style)

### 2.1 System Global Area (SGA)
- Bloque de memoria RAM compartido usado por una instancia de base de datos.
- Contiene caches y estructuras críticas para rendimiento:
  - Buffer cache (datos leídos/escritos)
  - Shared pool (parsed SQL, diccionario)
  - Redo log buffer
  - Large pool / Java pool / Streams pool
- Es la memoria central donde la base de datos guarda lo que está usando activamente.

---

## 3. Conceptos físicos de almacenamiento

1. **Data block**
   - Unidad mínima de almacenamiento que lee/escribe el DBMS.
   - Tiene: header + data + free space.

2. **Extent (Extensión)**
   - Conjunto contiguo de data blocks.
   - El DBMS asigna extensiones en lugar de bloques individuales para eficiencia.

3. **Segmento**
   - Conjunto de extensiones que pertenecen a un objeto (tabla, índice, etc.).
   - Representa el espacio físico ocupado por la tabla/índice dentro del tablespace.

---

## 4. Background processes

| Proceso                | Definición                                                                                                                                                                           |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| DBWR (Database Writer) | el proceso que **persiste en disco lo que ya fue modificado en memoria**.                                                                                                            |
| SMON (System Monitor)  | **Repara** y **mantiene** consistencia en la base de datos. Es el que se encarga de ayudar en caso de un apagón.                                                                     |
| LGWR (Log Writer)      | Guarda descripciones de los cambios que hubieron (redo-logs).                                                                                                                        |
| CKPT (Checkpoint)      | Sincroniza cabeceras de archivos. Se encarga de marcar un punto donde los datos ya deberían estar escritos en la base de datos. **Le dice al DBWR que persista los bloques sucios**. |
| PMON (Process Monitor) | Recoger lo que quedó inconsistente cuando un proceso muere. Gestiona procesos del sistema.                                                                                           |

---

## 5. Modos de servidor (Dedicated / Shared)

### 5.1 Dedicated

Corresponde a **un usuario por servidor**. Alto consumo de memoria. Simple.

### 5.2 Shared / MTS

Un **dispatcher** es el proceso que **recibe las conexiones de los clientes y gestiona la comunicación de entrada/salida**, sin ejecutar directamente las consultas.

---

## 6. Concurrencia y Consistencia Multiversión

### 6.1 Concurrencia

Es la habilidad de la base de datos de ejecutar multiples transacciones al mismo tiempo sin corromper datos. Se logra mediante locks, control de transacciones y niveles de aislamiento.

### 6.2 Multiversión (MVCC)

En lugar de sobrescribir datos, la base crea **versiones** de los mismos. Técnica que usa múltiples versiones de datos para permitir concurrencia sin bloqueos fuertes, manteniendo consistencia en las lecturas

---

## 7. Niveles de aislamiento en transacciones

- `Read Uncommitted` (Nivel más bajo)
  - Permite leer datos que aún no se han confirmado (dirty reads).
  - Ventaja: máxima concurrencia, mínimo bloqueo.
  - Riesgo: puede leer información que luego se deshace (no confiable).
  - Uso típico: análisis o reportes temporales donde no importa precisión instantánea.

- `Read Committed` (Nivel por defecto en Oracle)
  - Solo lee datos confirmados (no dirty reads).
  - Puede ver cambios de otras transacciones entre dos lecturas (non-repeatable reads).
  - Buen balance: cierto nivel de consistencia sin muchos bloqueos.
  - Uso típico: apps web, APIs, operaciones normales.

- `Repeatable Read`
  - Garantiza que los mismos SELECT dentro de la transacción devuelvan los mismos datos (no non-repeatable reads).
  - Aún puede aparecer phantom reads (nuevas filas insertadas por otros).
  - Uso típico: cálculos financieros intermedios, análisis con varias lecturas del mismo registro.

- `Serializable` (Nivel más alto)
  - Comportamiento equivalente a ejecutar transacciones una por una (serializadas).
  - Evita dirty reads, non-repeatable reads y phantom reads.
  - Genera más bloqueos y puede reducir concurrencia, pero asegura consistencia fuerte.
  - Uso típico: transferencias bancarias, contabilidad crítica, cuando la exactitud es prioritaria.

---

## 8. Resumen rápido comparativo

- Seguridad de lectura: Read Uncommitted < Read Committed < Repeatable Read < Serializable
- Concurrencia: Serializable < Repeatable Read < Read Committed < Read Uncommitted
- Nivel recomendado para OLTP general: `Read Committed`

- Un tablespace puede contener muchos segmentos.
- Un segmento contiene varias extensiones.
- Una extensión contiene muchas data blocks.
- El SGA es la memoria compartida, no un objeto en disco.
